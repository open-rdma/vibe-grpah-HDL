## Module Description:
A collection of specialized FIFO modules with unique capabilities beyond standard FIFOF, designed for the RDMA hardware implementation. This module provides three distinct FIFO variants:
1. mkScanFIFOF: A FIFO that supports a SCAN mode alongside normal FIFO
   operations. In scan mode, the FIFO iterates through its contents via a
   scan pointer while still allowing concurrent dequeue operations (but not
   enqueue). The module has three states: FIFO_MODE (normal operation),
   PRE_SCAN_MODE (snapshot item count and set scan pointer to dequeue
   pointer, allowing head modification), and SCAN_MODE (sequential scan
   output via scanPipeOut). Key behaviors: enqueue is blocked in scan mode;
   dequeue is allowed in both FIFO mode and scan mode (but not pre-scan
   mode); the scan head can be modified via the modifyHead method during
   pre-scan mode which overrides the scanned value for the first position.
   Implements full/empty tracking using item count registers, with
   assertions to enforce invariants. Uses mkCReg for conflict-free register
   access between methods and rules. The scanPipeOut produces elements one
   per cycle via the scanNext rule.

2. mkSearchFIFOF: A FIFO with a parallel search capability. Internally
   maintains a Vector of data registers with valid tags. The searchIfc
   provides a search method that takes a predicate function and uses
   findIndex to locate a matching item across all entries in a single cycle
   (combinational search). The FIFO itself uses an MSB-based full/empty
   detection scheme with enqPtr and deqPtr counters. The clear method resets
   all tags, enq/deq pointers, and empty/full flags. Uses mkCReg to allow
   concurrent access from enq, deq, and clear methods.

3. mkCacheFIFO2: A cache-like FIFO that supports pipelined search with
   configurable compare functions. The search is split into three pipeline
   stages: handleSearchReq (computes comparison results for all entries
   using a vectorized map), checkMapResult (passes results through a FIFO
   stage), and handleSearchResp (produces final Maybe result using
   compareResult2SearchResp conversion). The cache is implemented as a
   circular buffer with a write pointer (enqPtrReg). Push and clear use
   mkCReg for conflict-free method-to-rule communication. The
   genWorkComp4WorkReq function in WorkCompGen uses mkCacheFIFO2 to check
   for duplicate work completions. The compareFunc, checkCompareResult, and
   compareResult2SearchResp are higher-order function parameters that allow
   the caller to customize the multi-level comparison pipeline.

Interfaces defined: ScanCntrl (control scan operations), ScanFIFOF (combines FIFOF + ScanCntrl + scanPipeOut), SearchIfc2 (combinational search), SearchFIFOF (combines FIFOF + SearchIfc2), CacheIfc (cache insert/clear), SearchIfc (pipelined search with req/resp), CacheFIFO (combines CacheIfc + SearchIfc).
