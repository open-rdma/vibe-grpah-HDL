# 软件原始需求定义

你需要使用litegraph.js作为UI框架(npm install litegraph.js)，以Python语言作为服务端框架，开发一个可视化的数字电路设计工具，让用户在UI上完成多层级的模块设计蓝图，该蓝图会被后续用于指导大模型生成具体的RTL代码。首先使用brainsorm技能生成设计文档，然后按照设计文档完成项目设计。


## WebUI功能要求：
* 需要使用litegraph.js的subgraphs功能，实现对硬件RTL设计中的模块层级的展示和建模。
* 具备一个工具栏，可以快速在当前graph中添加subgraph，删除subgraph
* subgraph的连接端口可以自由添加、删除。每个端口要求有自己的名称和数据类型，以及其他可选的自定义属性。
    * 后续系统可能会内置添加一些固有的系统及属性，注意提前做好设计。
* 具备一个类型定义系统，可以预先定义出各种数据类型，在设置端口的属性时，可以引用这些预先定义好的数据类型。
* 如果端口的类型不一样，则禁止端口之间的连线。
* 具有工程的概念，一个工程是一套配置文件的全集，可以加载工程和保存工程。
* 数据文件的组织形式
    * 每个graph对应一个yaml数据文件（每个子graph当然也算作一个graph，对应一个子数据文件）
    * graph的层级结构对应到文件系统的目录结构，父图里可以引用子图的数据文件，由此串接形成整个图
    * 支持引用机制，即一个模块可以直接引用另一个子图的数据文件，作为自己的新的子图，当原始子图变更时，所有对应引用都等效变更。
        * 一个特例是“库”类型的模块，该类型模块对应的图可以存在一个特定的library目录下，由其他图多次引用。
        * 可以将整个工程定义为多个树形结构，其中，通常由两个树是系统默认存在的，一个叫做Top树，一个叫做Library树。
            * Top树用于对应传统RTL工程的TOP模块。
            * Library树用于存储被引用的模块。
            * 还可以由其他的树存在，而且上述Top和Library也是约定俗成，不是必须的。
    * 以git版本来做历史记录
* 图中的每个节点代表一个RTL模块。
    * 每个节点有自己的名称和自定义属性。
    * 每个节点要求有必须的功能描述字段，以及可选的测试方法字段。
        * 如果是叶子节点，则功能描述将被用于指导后续大模型生成最终的RTL模块代码
        * 如果是子图节点，则功能描述也阐述了子图之间信号的关系等，同样用于指导生成子图的RTL代码。不过子图节点内部的模块信息，也会被用于指导大模型最终生成RTL代码
        * 测试方法字段用于指导大模型生成对应模块的单元测试，描述激励信号的生成逻辑、checker逻辑等。
* 可以针对任意节点，调用大模型开始生成RTL，对对应的RTL输出做局部更新，也可以从Root开始，对整个工程进行彻底构建。
    * 支持全新构建，和增量构建两种。全新构建方式只给大模型发送当前模块所有属性和端口构成的提示词，不发送已经生成的代码；而增量模式，给大模型模块的所有属性和端口构成的提示词，以及之前生成的RTL代码。
* 这个项目类似一个“编译器”，可以借助大模型的能力，将用户给出的硬件描述翻译成用户指定的目标语言，而不是只能生成verilog
* WebUI应该有左侧或者顶部的工具栏，用以方便用户操作。
* litegraph.js的源代码在3rd/litegraph.js目录下，可以参考其中的doc进行开发。

改意见1：2.1节的Signals信号不能添加。内部信号应该是大模型在理解用户自然语言表述的意图后，自己生成的。如果用户对具体的实现方式有特殊要求，则应该在功能描述中进行细致描述，而不是要求用户提前想清楚内部信号如何设计。如果用户提前想清楚了内部的信号设计，那就是给用户添加了工作量，而削弱了大模型的主观能动性。你希望对每个信号明确其类型的出发点是好的，但用户不可能提前把这些都想清楚。我们希望用户可以在一个单独的文件中以自然语言描述需要用到的数据类型的大概样子（例如：一个包含foo、bar、baz信号的结构体，foo有16bit，表示一个浮点数，bar和baz都是2bit的整数），然后用户可以在其他的描述中引用这个数据类型的名字，然后由大模型决定如何使用这个自然语言描述生成目标代码。然而，在模块端口处，因为涉及到跨模块连线问题，需要做类型检查，因此端口应该由明确的数据类型表示。注意，这里提到的数据类型标识，都只是起到一个数据类型的标识ID的作用，而不是真实的数据类型。例如你在2.1小结给出的Bit#(TDiv#(reqNum,2))这样的类型表示，完全绑定了bluespec的语法，这是不对的。在我们的系统中，类型被定义为一段自然语言描述的“唯一ID”，而这段自然语言表述了真实类型的实际数据结构。在使用大模型生成RTL的时候，可以指定一个具有若干步骤的生成方式。例如第一步先扫描项目中所有的类型定义，然后先把类型定义通过大模型转换为目标RTL语言的类型定义形式，然后再开始生成其他RTL代码，在生成其他RTL代码的过程中就可以引用这些已经转换为目标语言RTL的类型定义了。因此，类型系统是一个非常重要的系统。需要特殊设计。

/Deli_AutoResearch  上一轮修改后的实时方案依然存在一些知识处理上的问题，而这个问题涉及到知识合并与最终代码生成两方面的问题。而这两个问题又是我们要解决的核心问题。因此，我们需要重新审视我们的整个目标，我们需要先解决如何设计一个“编译器”的问题，再解决如何提供图形界面展示的问题：

我们需要实现一种由大语言模型作为核心能力的“自然语言->RTL的编译器”，对于这个编译器而言，其输出目标语言是BluespecSystemVerilog、SystemVerilog、VHDL、Chisel等等语言，因此输出是什么样子的，已经相对确定了。 重要的问题是：
1. 这个编译器的输入数据是什么格式的
2. 这个编译器的内部运作流程是什么样子的

首先讨论第一个问题：
* 我们希望人类可以通过【嵌套多层级的图结构 + 自然语言描述】来定义出一个RTL项目的骨架和重要信息，然后由大模型根据这些信息来产出RTL代码 和 对应的测试用例代码。
  * 因此，我们需要核心定义在yaml文件中如何存储这些必要的描述信息。
  * 在设计存储结构的过程中，要考虑到这种结构在未来如何支持一个可视化的编辑工具（IDE）。
    * 这就像你可以使用最简单的文本编辑器来编辑各种yaml文件，但是对用户不友好。yaml是信息承载最基础的方式，但我们未来要在上面提供一个可视化的编辑环境，来通过拖拽等方式，构建出这样的yaml文件系统。
        * 目前，我们已经有了一个初步的UI可视化界面，可以参考其设计。

然后是第二个问题，基于大模型的编译器如何工作：
* 我们的设计是一个图结构，因此在生成RTL代码的过程中，应该可以通过图的连接关系，在当前模块的上游和下游节点之间进行遍历游走访问，例如上看2层，下看2层，将相邻2层内的知识、数据类型、接口信息、模块功能等，经过整理，给到当前模块生成代码。当然，生成的策略也有不同的方式，例如是从最开始的根节点，逐层向下生成，还是所有节点同时并行生成？这是需要进行实际实验对比，才可以得出效果的。因此，第二步工作的重点是如何把这个编译的流程控制好，也就是我们需要一个主控程序来按照一定的规则，完成图的遍历、知识的合并，然后调用Agent完成代码的生成。这个主控程序需要支持不同的策略，例如先遍历所有的类型定义，生成类型对应的RTL，再对图进行遍历，对于每个节点，都看一下相邻节点的信息，然后指导本节点生成RTL代码。

所以，我们现在需要将现有工程的backend、frontend两部分架构，变更为backend、frontend、compiler三部分架构，其中compiler是三部分中最重要的一个。这三个部分的关系如下：
    - compiler： 核心逻辑，负责把图+自然语言的yaml文件系统输入转换为目标语言的RTL代码。
    - frontend： 后续补充的逻辑，提供可视化的界面来生成compiler所需要的yaml文件系统
    - backend： 将compiler和frontend粘合起来的服务系统，对外提供静态frontend的网页服务，并响应用户在浏览器中的操作，管理yaml文件系统、调用compiler进行编译等。

基于上述背景知识，开始你在compiler上的自动调研、开发任务：
1、设计符合上述思想的yaml文件定义
2、设计符合上述思想的compiler实现
3、使用blue-rdma目录下的bluespec代码，依然按照 先将bluespec代码转写为yaml文件系统，再由yaml文件系统反过来编译生成新的bluespec代码，最后看生成bluespec代码是否可以通过testbench测试的流程，来自我评判正确性、并自我迭代。直到找到最优的yaml文件格式，以及最优的compiler实现。
4、注意：你设计的系统，其yaml文件系统中应该是抽象的，不与任何具体目标RTL语言特性绑定的。如果必须针对某种目标语言进行指引，应该将其写在某个层级（如目标语言模板、项目、图、端口、连线等各个层级）私有的知识字段中，而不是为这个语言特地增加某种字段。
